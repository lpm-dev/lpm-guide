# LPM Deploy Setup

Configure CI/CD and deployment platforms to install or publish lpm.dev packages. Keep this workflow focused on `.npmrc`, `LPM_TOKEN`, reproducible installs, publishing, and troubleshooting.

## Two Use Cases

| Role | Goal | Key Config |
| --- | --- | --- |
| **Consumer** | Install `@lpm.dev/*` packages during build | `.npmrc` + `LPM_TOKEN` |
| **Author** | Publish to lpm.dev from CI | `LPM_TOKEN` + publish workflow |

## Workflow

1. **Detect the situation** — platform, package manager, consumer vs author, exact error
2. **Generate `.npmrc`** — current default is scoped `@lpm.dev` registry config
3. **Configure secrets** — set `LPM_TOKEN` in the platform
4. **Set up caching** — prefer package-manager cache plus `~/.lpm/store` for LPM installs
5. **Add publish workflow** — only when the package is authored here
6. **Verify** — run a build and, for authors, a dry-run or checked publish

## `.npmrc`

Use the CLI when possible:

```bash
lpm setup ci npmrc
lpm setup local -d 30
```

Current generated shape:

```ini
//lpm.dev/:_authToken=${LPM_TOKEN}
@lpm.dev:registry=https://lpm.dev/api/registry/
```

This routes only `@lpm.dev/*` packages through lpm.dev. Leave the default npm registry alone unless the user explicitly asks for a custom registry setup.

### Placement

| Scenario | Location | Notes |
| --- | --- | --- |
| Single project | Project root `.npmrc` | Safe to commit when it contains `${LPM_TOKEN}`, not a literal token |
| Monorepo | Workspace root `.npmrc` | All workspaces inherit it |
| Local only | Project root `.npmrc` generated by `lpm setup local` | Contains a real read-only token; keep it out of git |

## Platform Secrets

Set `LPM_TOKEN` where the platform exposes build/runtime secrets:

| Platform | Location |
| --- | --- |
| GitHub Actions | Repository Settings → Secrets and variables → Actions |
| Vercel | Project Settings → Environment Variables |
| Netlify | Site Settings → Environment Variables |
| GitLab CI | Project Settings → CI/CD → Variables |

If the user does not have a token yet, run:

```bash
lpm login
lpm token-rotate
```

For local read-only `.npmrc` setup, prefer `lpm setup local -d <days>`.

## GitHub Actions

Consumer workflow:

```yaml
name: Build
on: [push, pull_request]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - run: lpm ci --offline --strict-integrity
        env:
          LPM_TOKEN: ${{ secrets.LPM_TOKEN }}

      - run: lpm run build
      - run: lpm test
```

Author publish workflow:

```yaml
name: Publish
on:
  push:
    tags: ["v*"]

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - run: lpm ci --offline --strict-integrity
        env:
          LPM_TOKEN: ${{ secrets.LPM_TOKEN }}

      - run: lpm publish --provenance --min-score 70
        env:
          LPM_TOKEN: ${{ secrets.LPM_TOKEN }}
```

If the project has not adopted `lpm ci`, use the existing package manager install command, but keep `.npmrc` and `LPM_TOKEN` in place for `@lpm.dev/*` packages.

## Caching

Cache the package manager's cache and, for LPM installs, `~/.lpm/store`. Key by OS and lockfile hash.

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.lpm/store
    key: lpm-store-${{ runner.os }}-${{ hashFiles('lpm.lock', 'lpm.lockb') }}
    restore-keys: |
      lpm-store-${{ runner.os }}-
```

Avoid recommending `node_modules` caching as the default. It is more fragile than store/cache-based installs and often conflicts with lockfile-driven reproducibility.

## Verify

Check:

1. `.npmrc` contains `@lpm.dev:registry=https://lpm.dev/api/registry/`
2. `LPM_TOKEN` is set in the platform
3. The build can install `@lpm.dev/*` packages
4. Authors can run `lpm publish --check` or the configured publish workflow

## Troubleshooting

| Error | Likely Cause | Fix |
| --- | --- | --- |
| `401 Unauthorized` | `LPM_TOKEN` missing, expired, or not visible to the build step | Set/refresh `LPM_TOKEN`; verify with `lpm whoami` locally |
| `404 Not Found` for `@lpm.dev/*` | Missing scoped registry config or package is not visible to this token | Run `lpm setup ci npmrc` and confirm package access |
| `ERR_SCOPE_NOT_FOUND` | npm/pnpm/yarn did not read `.npmrc` | Put `.npmrc` at project/workspace root and confirm build working directory |
| `ERESOLVE` / version conflict | Dependency graph conflict | Use committed lockfile and run the same install command locally |
| Token works locally but not in CI | Different secret, branch/environment scoping, or masked variable typo | Re-check platform secret scope and variable name |

## Important Notes

- Never hardcode tokens in `.npmrc`, Dockerfiles, workflow YAML, or app source
- Commit `.npmrc` only when it uses `${LPM_TOKEN}`
- Keep real local `.npmrc` tokens out of git
- MCP servers and Agent Skills are separate AI integration mechanisms; do not add MCP setup to deployment guidance unless the user specifically asks
